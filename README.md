# PG-microservice (game-service)

![Java Version](https://shields.io)
![Spring Boot](https://shields.io)
![gRPC](https://shields.io)

Микросервис обработки и хранения игровой статистики на базе СУБД **PostgreSQL**. Сервис является аналитическим ядром системы: он рассчитывает рекорды, ведет учет игровых сессий, подсчитывает количество попыток и предоставляет внутренний gRPC-интерфейс для игрового движка, а также REST-эндпоинты для администрирования.

## 🛠️ Стек технологий и особенности

* **Язык и производительность:** Java 21 со включенными виртуальными потоками (Virtual Threads). Благодаря кастомной конфигурации `GrpcServerConfigurer`, обработка входящих gRPC-запросов полностью переведена на легковесные потоки Loom, минуя ограничения пула ОС-потоков.
* **Фреймворк:** Spring Boot 3.4.2 + Spring Cloud 2024.0.0 (Netflix Eureka Client).
* **База данных:** PostgreSQL + Spring Data JPA + Управление транзакциями (`@Transactional`).
* **Оптимизация под Loom и Batching:** Пул соединений HikariCP увеличен до **50** соединений. Стратегия генерации ID переведена с `IDENTITY` на высокопроизводительный `SEQUENCE` (алгоритм HiLo с `allocationSize=50`) со включенной пакетной записью (`jdbc.batch_size`). Это позволяет инстансам выделять пулы уникальных идентификаторов в памяти Java, полностью устраняет сетевые задержки при одиночных вставках и исключает блокировки СУБД при лавинообразном росте параллельных виртуальных потоков.

* **Асинхронная параллелизация:** Логика метода `GetGameScore` оптимизирована под особенности Project Loom — сбор четырех различных метрик из базы данных и кэша запускается параллельно через `CompletableFuture.supplyAsync`, что снижает Latency ответа до минимума.
* **Защита продакшена (Fail-safe выборки):** Метод `GetAllGames` аппаратно ограничен на уровне СУБД конструкцией `LIMIT 100`. Это гарантирует стабильную работу фронтенда без изменения gRPC-контракта, полностью исключает утечки памяти (OutofMemoryError) и защищает кэш от переполнения при миллионах записей в БД.
* **Агрегация на стороне СУБД:** Расчет максимумов (`MAX`) и количества попыток (`COUNT`) делегирован самой базе данных на уровне легковесных SQL-запросов вместо тяжелой фильтрации через Java Streams.
* **Кэширование:** Высокоэффективный локальный кэш Caffeine для моментальной отдачи рекордов и максимальной разгрузки процессора БД.

## 🔀 Сетевые интерфейсы и Порты

Микросервис одновременно слушает два сетевых порта:

* **HTTP Порт (2222):** Предназначен для административных REST-запросов и мониторинга (Actuator).
* **gRPC Порт (6566):** Выделенный внутренний канал для высокоскоростного общения с игровым движком Тетриса. Настройки порта автоматически публикуются в реестре Эврики через метаданные.

## 🔌 API Endpoints (HTTP REST)

* `GET/POST /delete?playerName={имя}` — Административный эндпоинт, запускающий каскадное удаление всей истории игр и статистики конкретного пользователя из PostgreSQL с параллельной очисткой кэша.

## 🛰️ Контракт gRPC Сервиса (GameIntegrationService)

Внутренний интерфейс взаимодействия, описанный в Protobuf, предоставляет игровому движку следующие высокоскоростные RPC-методы:

1. **`GetGameScore` (Принимает `PlayerRequest`, возвращает `GameScoreResponse`):**
    * Выгружает комплексный отчет: имя абсолютного чемпиона системы, исторический рекорд проекта, личный рекорд запрашивающего игрока и общее число его сыгранных матчей.
2. **`GetAllGames` (Принимает `Empty`, возвращает `AllGamesResponse`):**
    * Возвращает полный список всех сыгранных в системе матчей (ID, имя, счет) с использованием кастомного нативного SQL-запроса.
3. **`DoRecord` (Принимает `GameRecordRequest`, возвращает `Empty`):**
    * Фиксирует результат завершенной игры. Метод конвертирует входящие Protobuf-данные в JPA-сущность `Game` и сохраняет её в базу данных.

## ⚡ Умное точечное кэширование (Caffeine)

Конфигурация кэша ограничена 100 записями со временем жизни в 5 минут. Чтобы действия одного игрока не сбрасывали статистику всей системы, настроены изолированные составные ключи:
* **`best_result`** — Глобальный кэш для лучшего игрока (`key: 'player'`) и лучшего счета (`key: 'score'`). Сбрасывается только при фиксации нового рекорда.
* **`games`** — Кэширует общий топ лучших результатов (`key: 'topRecords'`).
* **`player_stats`** — Точечный персональный кэш игрока. Генерирует ключи с суффиксами вида `{имя_игрока}_best` и `{имя_игрока}_attempts`. При добавлении или удалении матча сбрасывается кэш только этого конкретного пользователя.

## 🚀 Запуск проекта

### Требования
* Установленный JDK 21
* Запущенная СУБД PostgreSQL с базой данных `game`
* Активный `Eureka-microservice` на порту 1111

### Порядок запуска
1. Перейти в каталог проекта `PG-microservice`.
2. Убедиться, что в PostgreSQL создана чистая база данных `game`. При первом старте Hibernate автоматически выполнит DDL-скрипты, создаст таблицу `games` и необходимую последовательность `games_sequence` для генерации ID пачками по 50 штук.
3. Запустить приложение с помощью Maven сборщика. Проект автоматически скомпилирует `.proto` файлы и развернет gRPC-сервер на порту 6566.
4. **Важно для диагностики Loom:** При запуске JAR-файла рекомендуется передавать JVM-аргумент `-Djdk.tracePinnedThreads=short`. Это позволит контролировать отсутствие Thread Pinning (блокировки несущих ОС-потоков) в критических секциях Hibernate и драйвера СУБД при работе с виртуальными потоками.
